# 每日盘前与盘后股票研究设计

## 目标

在中国时间的每个工作日生成两份研究报告：09:00 的盘前报告与 23:00 的盘后报告。盘后报告必须在可核验的前提下使用当天已完成的 A 股、北交所和港股收盘日线，并继续通过既有 `generate` 链路发送完整报告到飞书。

## 设计

`DailyRunRequest` 与 `DailyReport` 增加可选 `run_slot`：`pre_market`、`post_market` 或 `null`。`null` 保持手动和历史 JSON 的原有行为与路径兼容；两个自动化任务始终填写明确槽位。

盘前和未指定槽位的 `market_sessions.completed_session` 必须早于 `report_date`。盘后槽位可等于 `report_date`，但只有市场未休市时才允许；休市市场必须继续使用更早的已完成交易日。`ReportBuilder` 已依照请求会话日期拉取和校验行情，因此无须改变指标或建议规则。

带运行槽位的报告分别写入 `<app-home>/reports/YYYY-MM-DD/pre-market/` 与 `<app-home>/reports/YYYY-MM-DD/post-market/`，防止同日报告互相覆盖。历史和手动报告仍使用原有 `<app-home>/reports/YYYY-MM-DD/report.*` 路径。SQLite 的日期级最新报告保持现有接口，23:00 运行会成为该日期的最新版本；盘前路径仍由运行记录完整保留。

## 自动化与报告口径

保留现有工作日 09:00 自动化并将其显式标为盘前。新增工作日 23:00 自动化，两者均在本地项目环境执行并仅在失败时由 Codex App 提醒。盘后提示词要求：确认每个市场当天已收市；以当日收盘、涨跌、成交量、技术指标、公告与盘中/盘后消息作为核心；未能确认当天完成日线时必须标明数据缺口并退回最后可验证会话，绝不把盘中数据当作收盘。

两次运行都读取同一持久化股票配置、生成并检查 JSON/Markdown/HTML，并通过已有 `stock-research generate` 的飞书通知逻辑发送完整报告；不新增券商、交易或密钥能力。

## 验收

- 盘后请求可接受未休市市场的同日完成会话，盘前与闭市市场仍拒绝同日会话。
- 同一日期的盘前和盘后报告写入不同目录；旧路径仍可用于未标注槽位的报告。
- 自动化提示词清楚区分两种模式，明确 09:00、23:00 和盘后收盘复盘要求。
- 已存在 09:00 自动化更新为盘前模式，并创建工作日 23:00 盘后自动化。
